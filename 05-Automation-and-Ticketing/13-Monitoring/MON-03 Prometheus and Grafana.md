# MON-03: Prometheus and Grafana

> [!abstract] Overview
> *Prometheus aur Grafana modern IT infrastructure mein observability ki back-bone hain.* Prometheus metrics collect karta hai (pull model) aur alert generate karta hai, jabki Grafana un metrics ko sundar aur samajhne laayak dashboards mein visualize karta hai. Ek L1/L2 engineer ke liye in tools ka gyaan hona bohot zaroori hai taaki server, application ya network ka issue hone par fauran detect aur troubleshoot kiya ja sake.

---
## 🧠 Concept Overview

- **What it is** — Prometheus is an open-source systems monitoring and alerting toolkit originally built at SoundCloud. It records real-time metrics in a time series database. Grafana is a multi-platform open-source analytics and interactive visualization web application.
- **Why it matters** — Enterprise environments mein hazaron servers aur microservices hoti hain. *Agar koi server down hota hai ya CPU 100% ho jata hai, toh humein manually check karne ki zaroorat nahi padti, Prometheus usko detect karke Grafana par red alert dikha deta hai.*
- **Where you see this** — Used everywhere in cloud-native environments, Kubernetes clusters, on-premise data centers for monitoring servers, databases (like MySQL, PostgreSQL), network devices, and applications.

**L1 / L2 / L3 Split:**

| 👨‍💻 Level | 📋 Responsibility |
|---------|-----------------|
| **L1** | Grafana dashboards check karna, red alerts dekhna, aur basic service status verify karna. |
| **L2** | Prometheus exporters install karna, basic PromQL queries likhna, aur alerts configure karna. |
| **L3** | Architecture design karna, High Availability (HA) setup, Thanos/Cortex integration, custom metrics aur complex PromQL optimization. |

> [!tip] Seedha Simple Mein
> *Prometheus ek CCTV camera aur recorder ki tarah hai jo har ek server par nazar rakhta hai aur data record karta hai. Grafana woh TV screen hai jahan security guard (Admin) baith kar sab kuch ek saath dekhta hai.*

## 🔬 Technical Deep Dive

### 1. Prometheus Architecture & Components

> [!info] Key Concept
> Prometheus works on a **Pull Model** over HTTP, scraping metrics from configured endpoints.

Prometheus ecosystem consists of several main components:
- **Prometheus Server:** Scrapes and stores time-series data. *Yeh core engine hai jo data pull karke apne paas save karta hai.*
- **Client Libraries:** For instrumenting application code.
- **Pushgateway:** For supporting short-lived jobs. *Jo jobs jaldi khatam ho jaati hain aur pull hone ka wait nahi kar sakti, woh Pushgateway ko data bhejti hain.*
- **Exporters:** Special services for exporting existing metrics from third-party systems like HAProxy, StatsD, Linux OS (Node Exporter).
- **Alertmanager:** Handles alerts generated by Prometheus and routes them to email, Slack, PagerDuty, etc. *Jab bhi threshold cross hota hai, Alertmanager hi email ya message bhejta hai.*

> [!danger] Common Mistake
> Forgetting to open the firewall port for Exporters (e.g., port 9100 for Node Exporter). *Agar Prometheus ko exporter ka port accessible nahi hoga, toh woh target 'DOWN' dikhayega.*

### 2. Time-Series Data and PromQL

Prometheus stores data as time series: streams of timestamped values belonging to the same metric and the same set of labeled dimensions.
- **Metric Name:** e.g., `http_requests_total`
- **Labels:** e.g., `method="POST"`, `handler="/api/tracks"`

**PromQL (Prometheus Query Language):**
Used to extract and manipulate time-series data.
- Instant Vector: Returns a single value per time series. e.g., `node_cpu_seconds_total`
- Range Vector: Returns a set of values over time. e.g., `node_cpu_seconds_total[5m]`

### 3. Grafana Integration

Grafana connects to Prometheus as a Data Source.
- **Dashboards:** Created using panels.
- **Panels:** Each panel uses a PromQL query to fetch data from Prometheus.
- **Variables:** Allows dynamic dashboards where you can select specific servers or environments from a dropdown.

## 🛠️ Step-by-Step Lab

> [!warning] Pre-requisites
> - A Linux server (RHEL/Ubuntu)
> - Root or sudo privileges
> - Internet connection to download packages

### Step 1: Install Node Exporter (on Target Server)

Node exporter exposes hardware and OS metrics. *Sabse pehle jis server ko monitor karna hai, wahan Node Exporter dalenge.*

```bash
# Download node exporter
wget https://github.com/prometheus/node_exporter/releases/download/v1.6.1/node_exporter-1.6.1.linux-amd64.tar.gz

# Extract the archive
tar xvfz node_exporter-*.*-amd64.tar.gz

# Move the binary to /usr/local/bin
sudo mv node_exporter-*.*-amd64/node_exporter /usr/local/bin/

# Create a systemd service file
sudo vi /etc/systemd/system/node_exporter.service
```

Add the following to the service file:
```ini
[Unit]
Description=Node Exporter
After=network.target

[Service]
User=node_exporter
Group=node_exporter
Type=simple
ExecStart=/usr/local/bin/node_exporter

[Install]
WantedBy=multi-user.target
```

```bash
# Start and enable the service
sudo systemctl daemon-reload
sudo systemctl start node_exporter
sudo systemctl enable node_exporter
```

> [!success] Expected Output
> ```
> Active: active (running) since ...
> ```
> Port 9100 par ab metrics expose ho rahi hongi. (Check via `curl localhost:9100/metrics`)

### Step 2: Configure Prometheus to Scrape Node Exporter

Prometheus server pe `prometheus.yml` ko edit karein.

```yaml
# Add a new job under scrape_configs
scrape_configs:
  - job_name: 'linux_servers'
    static_configs:
      - targets: ['<TARGET_SERVER_IP>:9100']
```

```bash
# Restart Prometheus
sudo systemctl restart prometheus
```

### Step 3: Add Prometheus as Data Source in Grafana

1. Open Grafana UI (usually `http://<GRAFANA_IP>:3000`).
2. Go to **Connections** > **Data Sources** > **Add data source**.
3. Select **Prometheus**.
4. Set the HTTP URL to `http://<PROMETHEUS_IP>:9090`.
5. Click **Save & Test**. *Agar connection successful hai, toh green tick aayega.*

### Step 4: Import a Grafana Dashboard

1. Go to Dashboards > Import.
2. Enter ID `1860` (Popular Node Exporter Full dashboard).
3. Click Load, select your Prometheus data source, and click Import.

## 🚑 Troubleshooting Guide

| ⚠️ Problem | 🔍 Wajah (Cause) | 🛠️ Fix |
|-----------|----------------|-------|
| Target shows `DOWN` in Prometheus | Exporter is not running, or firewall is blocking the port. | Check exporter service. `telnet <IP> <PORT>` to verify network connectivity. |
| Grafana shows "Data Source Error" | Prometheus service is down, or wrong IP/Port in Grafana config. | Verify Prometheus is running on port 9090. Check Grafana Data Source URL. |
| High Memory/CPU usage by Prometheus | Too many metrics (high cardinality) or retention period is too long. | Drop unnecessary labels, use `recording rules`, or reduce `storage.tsdb.retention.time`. |
| Alerts are not firing | Alertmanager is not reachable, or alert rules are misconfigured. | Check Prometheus `alerting` config and `promtool check rules` to validate rule syntax. |

## 🎫 Real-World Ticket Scenarios

### 🎫 Scenario 1: Critical Target Down Alert

> [!example] Ticket
> "Alert received: `InstanceDown` for `DB-Server-01:9100`. Grafana dashboard shows no data for the last 15 minutes."

**L1 Response:** Check if the DB-Server-01 is pingable and reachable via SSH. *Pehle dekho server khud zinda hai ya nahi.*
**Escalation Trigger:** Server is up, but Node Exporter is failing to start even after manual restart.
**L2 Resolution:** SSH into DB-Server-01, check `journalctl -u node_exporter`. Found out disk is 100% full, preventing exporter from writing basic logs or functioning. Cleared up `/var/log` space and restarted the exporter.

### 🎫 Scenario 2: Grafana Dashboard Showing "No Data"

> [!example] Ticket
> "Developer team reported that the 'Kubernetes Cluster Health' Grafana dashboard is empty and panels show 'No Data'."

**L1 Response:** Verify if other dashboards are working. Check if Prometheus Data Source is returning success in Grafana.
**Escalation Trigger:** Data source test passes, but the specific PromQL queries are returning empty results.
**L2/L3 Resolution:** Checked Prometheus server UI and noticed the `kube-state-metrics` target was accidentally removed from `prometheus.yml` during a recent automation run. Reverted the config change and reloaded Prometheus.

### 🎫 Scenario 3: Alert Fatigue (Too Many Irrelevant Alerts)

> [!example] Ticket
> "NOC team is complaining about receiving 500+ CPU warning emails per day for non-production servers."

**L1 Response:** Verify the alert name and source in Alertmanager. Confirm these are indeed Dev/Test servers.
**Escalation Trigger:** Need to modify the alert routing configuration.
**L2 Resolution:** Updated Alertmanager `alertmanager.yml` to route alerts with label `env="dev"` to a separate Slack channel instead of the primary NOC email, and grouped alerts by `instance` to reduce noise. *Dev/Test ke alerts ko email ki jagah Slack me bhej diya.*

## 🎤 Interview Questions

> [!question] Q1: What is the difference between Pull and Push based monitoring?
> **Answer:** In a **Pull** model (like Prometheus), the central monitoring server connects to target machines to 'scrape' or fetch metrics. In a **Push** model (like InfluxDB/Telegraf or Nagios passive checks), the target machines proactively send their data to the central server.

> [!question] Q2: How do you monitor a short-lived batch job in Prometheus if it uses a Pull model?
> **Answer:** We use the **Prometheus Pushgateway**. The short-lived job pushes its metrics to the Pushgateway before it exits. Prometheus then scrapes the Pushgateway at regular intervals to collect those metrics.

> [!question] Q3: What is PromQL and what is the difference between an Instant Vector and a Range Vector?
> **Answer:** PromQL is the Prometheus Query Language. An **Instant Vector** returns the single latest value for a metric (e.g., current CPU usage). A **Range Vector** returns a set of values over a specified time duration (e.g., CPU usage over the last 5 minutes, usually denoted by `[5m]`), which is useful for calculating rates or averages.

> [!question] Q4: How does Alertmanager handle alert grouping?
> **Answer:** Alertmanager can group similar alerts into a single notification. For example, if a network switch goes down and 100 servers behind it become unreachable, Alertmanager groups those 100 `InstanceDown` alerts into one single email, preventing a flood of notifications (alert storm).

==**Exam Tip:** Remember that Prometheus stores data locally in a TSDB (Time Series Database) and is not inherently designed for long-term distributed historical storage out-of-the-box. For long-term storage, solutions like Thanos or Cortex are integrated.==
