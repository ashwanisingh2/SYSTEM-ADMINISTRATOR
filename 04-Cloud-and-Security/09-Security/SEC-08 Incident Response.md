# SEC-08: Incident Response (IR)

> [!abstract] Overview
> *Incident Response (IR) ek structured approach hai kisi cyber attack ya security breach ko handle karne ka.* The goal is to identify the attack, contain the damage, eradicate the threat, and recover normal operations as quickly as possible. Ek L1/L2 support engineer ya sysadmin ke liye IR ka process samajhna bahut zaroori hai kyunki first line of defense wahi hote hain jo suspicious activity report karte hain.

---
## 🧠 Concept Overview

- **What it is** — A structured methodology for handling security incidents, breaches, and cyber threats. It follows frameworks like NIST (National Institute of Standards and Technology) or SANS.
- **Why it matters** — *Jab system compromise hota hai, toh har second keemat rakhta hai. Ek well-defined process damage aur data loss ko minimize karta hai.*
- **Where you see this** — Handling ransomware attacks, compromised user accounts, DDOS attacks, malware infections, and unauthorized access.

**L1 / L2 / L3 Split:**

| 👨‍💻 Level | 📋 Responsibility |
|---------|-----------------|
| **L1** | Identify and report. User issues ko analyze karna (e.g., user can't login, unusual pop-ups). Raise ticket to SOC/Security team. *First level investigation karna aur escalate karna.* |
| **L2** | Containment and initial triage. Isolating affected machines from the network, disabling compromised accounts, running basic forensic data collection (logs). *Machine ko network se alag karna aur logs collect karna.* |
| **L3 / SOC** | Deep forensic analysis, malware reverse engineering, threat eradication, post-incident reporting, and strategic recovery. *Root cause nikalna aur system ko completely clean karna.* |

> [!tip] Seedha Simple Mein
> *Jab building mein aag lagti hai, toh aap pehle fire alarm bajate ho (L1), phir fire extinguisher se aag bujhane ki koshish karte ho ya darwaze band karte ho taaki aag na phaile (L2), aur finally fire brigade aati hai aag ko completely bujhane aur root cause check karne (L3). Incident Response exactly yahi hai, bas computers ke liye.*

## 💡 Real-World Analogy

> [!info] Think of it like this...
> **Crime Scene Investigation** is like **Incident Response** because...
>
> - **Preparation:** Police is always trained and has equipment ready. (Securing backups, training staff)
> - **Identification:** Someone calls 100 to report a crime. (Alert generated by SIEM or user)
> - **Containment:** Police puts up yellow tape to secure the scene. (Isolating the infected server)
> - **Eradication:** Arresting the criminal. (Removing malware/hacker access)
> - **Recovery:** Cleaning up the area and reopening it to the public. (Restoring servers from backup)
> - **Lessons Learned:** Analyzing how the crime happened to prevent it next time. (Updating firewall rules)

## 🔬 Technical Deep Dive

### 1. The Incident Response Life Cycle (NIST Framework)

> [!info] Key Concept
> The NIST (National Institute of Standards and Technology) framework defines four core phases of Incident Response.

**Phase 1: Preparation**
Establishing policies, procedures, and training.
- Ensuring logs are enabled (Syslog, Event Viewer).
- Having backups in place (immutable backups).
- Creating an Incident Response Plan (IRP).
- *Attack hone se pehle ki tayyari karna.*

**Phase 2: Detection and Analysis**
Identifying that an incident has occurred.
- SIEM (Security Information and Event Management) alerts.
- User reports (e.g., "My files are locked").
- Analyzing packet captures, log files, and memory dumps to verify the threat.
- *Pata lagana ki attack hua hai aur kitna bada hai.*

**Phase 3: Containment, Eradication, and Recovery**
- **Containment:** Stopping the spread. (e.g., unplugging the network cable, blocking IP on firewall).
- **Eradication:** Removing the root cause. (e.g., deleting malware, patching the vulnerability).
- **Recovery:** Restoring operations. (e.g., restoring from backups, bringing systems online).
- *Attack ko rokna, virus ko delete karna, aur system ko wapas normal karna.*

**Phase 4: Post-Incident Activity (Lessons Learned)**
- Reviewing what happened.
- Updating policies and procedures.
- Writing a detailed Incident Report.
- *Galti se seekhna taaki future mein aisi mistake na ho.*

### 2. Digital Forensics Basics

> [!info] Order of Volatility
> During an investigation, data must be collected based on its volatility (how easily it can be lost when powered off).

1. CPU Registers and Cache
2. RAM (Routing tables, ARP cache, process tables, kernel statistics)
3. Swap Space / Paging File
4. Hard Disk Drive (Files, logs)
5. Remote Logs (Syslog)
6. Backups

> [!danger] Common Mistake
> Never power off a compromised machine immediately. *System ko direct shutdown mat karo, isse RAM ka data (evidence) delete ho jayega. Sirf network se disconnect karo.*

## 🛠️ Step-by-Step Lab

> [!warning] Pre-requisites
> - A compromised Linux or Windows VM.
> - Admin/root access.
> - Basic knowledge of CLI tools.

### Step 1: Identification (Check Network Connections)

*Sabse pehle check karo kaunse suspicious connections establish hue hain.*

```bash
# Windows
netstat -ano | findstr ESTABLISHED

# Linux
ss -tupan | grep ESTAB
```

> [!success] Expected Output
> You will see a list of IP addresses connected to the system. Look for strange external IPs.

### Step 2: Containment (Isolate the Machine)

*Machine ko network se alag karna bina power off kiye.*

```bash
# Windows (Disable Network Adapter)
Disable-NetAdapter -Name "Ethernet0" -Confirm:$false

# Linux (Bring interface down)
sudo ip link set eth0 down
```

### Step 3: Analysis (Check Running Processes)

*Suspicious background process ko dhoondhna.*

```bash
# Windows
Get-Process | Sort-Object CPU -Descending | Select-Object -First 10

# Linux
top -c
ps aux --sort=-%cpu | head -n 10
```

### Step 4: Eradication (Kill malicious process)

*Malicious process ko permanently kill karna.*

```bash
# Windows (Replace 1234 with PID)
Stop-Process -Id 1234 -Force

# Linux (Replace 1234 with PID)
sudo kill -9 1234
```

### Step 5: Recovery (Restore from Backup or Clean State)

*Agar system corrupt hai, toh usko wipe karke backup se restore karna padega.*
(This step usually involves enterprise backup solutions like Veeam or Commvault).

## ⌨️ Command Cheat Sheet

| ⌨️ Command | 🛠️ Kya karta hai | 📝 Example |
|-----------|-----------------|-----------|
| `netstat -ano` | Active network connections check karta hai (Win) | `netstat -ano \| findstr :443` |
| `ss -tupan` | Active network sockets aur process IDs dikhata hai (Linux) | `ss -tupan \| grep ESTAB` |
| `Get-Process` | Running processes ki list (PowerShell) | `Get-Process \| Sort CPU -Desc` |
| `ps aux` | All running processes (Linux) | `ps aux \| grep suspicious_name` |
| `taskkill` | Windows process ko force kill karta hai | `taskkill /F /PID 1234` |
| `kill -9` | Linux process ko forcefully terminate karta hai | `kill -9 5678` |
| `iptables` | Linux firewall rules manage karta hai (IP block/allow) | `iptables -A INPUT -s 1.2.3.4 -j DROP` |
| `New-NetFirewallRule` | Windows firewall rule create karta hai | `New-NetFirewallRule -DisplayName "Block IP" -Direction Inbound -Action Block -RemoteAddress 1.2.3.4` |

## 🚑 Troubleshooting Guide

| ⚠️ Problem | 🔍 Wajah (Cause) | 🛠️ Fix |
|-----------|----------------|-------|
| Cannot login to Server (Admin locked out) | Attacker changed credentials or brute-force triggered lockout. | Access via Out-of-Band management (iLO/iDRAC) or console, reset password via Local Admin. |
| Server CPU is at 100% | Crypto-mining malware is running in the background. | Identify PID using `top` or `Task Manager`, kill the process, and delete the executable file. |
| Files have `.encrypted` extension | Ransomware infection. | Disconnect from network immediately. Do NOT pay ransom. Restore data from offline/immutable backups. |
| Suspicious outgoing traffic on port 4444 | Reverse shell connection to a C2 (Command & Control) server. | Block the destination IP on the perimeter firewall. Run antivirus/EDR scan to find the payload. |

## 🎫 Real-World Ticket Scenarios

### 🎫 Scenario 1: Ransomware Infection Alert

> [!example] Ticket
> "Multiple users are reporting that they cannot open their Excel files on the shared drive. The files have a `.lock` extension and there is a `READ_ME.txt` file in the folder."

**L1 Response:** Immediately escalate to L2/Security team. *Users ko bolna system band na karein, network cable nikal dein.*
**Escalation Trigger:** Confirmed ransomware symptoms (encrypted files, ransom note).
**L2/L3 Resolution:**
1. Identify the 'Patient Zero' (the user who got infected first by checking file owner of the encrypted files).
2. Isolate Patient Zero's machine and the File Server from the network.
3. Review logs to identify the initial infection vector (e.g., Phishing email).
4. Restore the File Server data from the previous night's immutable backup.
5. Re-image Patient Zero's laptop.

### 🎫 Scenario 2: Compromised User Account

> [!example] Ticket
> "Azure AD Identity Protection alert: Impossible travel activity for user 'john.doe'. Logins detected from New York and Moscow within 10 minutes."

**L1 Response:** Contact John Doe via phone to verify if he is traveling or using a VPN. *User se confirm karna ki usne login kiya hai ya nahi.*
**Escalation Trigger:** User confirms he is in New York and not using a VPN.
**L2/L3 Resolution:**
1. **Containment:** Force password reset for John Doe's account. Revoke all active Azure AD refresh tokens (`Revoke-AzureADUserAllRefreshToken`).
2. **Analysis:** Review sign-in logs to see what data the attacker accessed (SharePoint, Exchange).
3. **Eradication:** Ensure MFA is strictly enforced and remove any malicious inbox forwarding rules.

### 🎫 Scenario 3: DDoS Attack on Web Server

> [!example] Ticket
> "Website is down. SolarWinds shows 100% network bandwidth utilization on the external interface of the Web Application Firewall (WAF)."

**L1 Response:** Verify website accessibility using external tools. Check WAF monitoring dashboard. *Confirm karna ki attack actual mein ho raha hai.*
**Escalation Trigger:** Volume of traffic exceeds normal capacity by 1000x, causing denial of service.
**L2/L3 Resolution:**
1. Coordinate with ISP to implement 'Blackhole routing' or rate limiting.
2. Enable DDoS protection services (e.g., Cloudflare 'Under Attack' mode or Azure DDoS Protection).
3. Analyze traffic patterns to create specific WAF rules blocking the attack signatures.

## 🎤 Interview Questions

> [!question] Q1: What is the difference between an Event and an Incident?
> **Answer:** An **Event** is any observable occurrence in a system or network (e.g., a user logging in, a firewall blocking a connection). An **Incident** is an event that negatively affects the confidentiality, integrity, or availability of an information system (e.g., a malware infection, unauthorized access). *Har incident event hota hai, par har event incident nahi hota.*

> [!question] Q2: If you suspect a machine is infected with malware, should you pull the power plug immediately? Why or why not?
> **Answer:** No, you should **NOT** pull the power plug immediately. Doing so will destroy volatile memory (RAM), which may contain crucial forensic evidence like active network connections, running malicious processes, or decryption keys. You should disconnect the network cable instead to contain the threat.

> [!question] Q3: What is the purpose of the 'Lessons Learned' phase in Incident Response?
> **Answer:** The 'Lessons Learned' or Post-Incident Activity phase is crucial for identifying what went wrong, evaluating the effectiveness of the response, and updating security policies, procedures, and controls to prevent similar incidents in the future.

> [!question] Q4: What is the 'Order of Volatility' in digital forensics?
> **Answer:** The Order of Volatility is the sequence in which digital evidence should be collected, starting with the most fragile data. It goes from CPU registers/cache -> RAM -> Swap/Paging files -> Hard Drives -> Remote Logs -> Backups.

==**Exam Tip:** Always remember that Containment comes BEFORE Eradication. You must stop the bleeding before you try to cure the disease!==
