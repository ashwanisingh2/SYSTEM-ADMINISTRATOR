# Event Viewer

---

## Concept Overview
- **What it is**: Windows Event Viewer is a centralized logging utility that records system, security, application, and service events, warnings, and errors generated by the Windows operating system and third-party software.
- **Why it matters for a support engineer**: Troubleshooting system errors, security audits, and application crashes without logs is impossible. A support engineer must know how to filter log databases, locate specific Event IDs, create custom views, and export logs to diagnose problems.
- **Where you encounter this in real job**: Finding the cause of random restarts, identifying account lockout sources, auditing failed logins, diagnosing application crashes, and monitoring device driver errors.
- **L1 vs L2 vs L3 responsibility split**:
  - **L1**: Reviews local Application and System logs, checks basic event descriptions, and filters logs by Event IDs.
  - **L2**: Creates custom XML log views, configures Event Subscriptions (Log forwarding), and analyzes crash dump logs.
  - **L3**: Establishes centralized logging systems (SIEM/Log Analytics), writes custom event collection scripts, and audits corporate security compliance logs.

## Technical Deep Dive

### 1. The Core Log Categories
Windows divides events into several primary channels:
- **Application Log**: Records events logged by local applications and programs (e.g., database connection drops or program crashes).
- **Security Log**: Records security audit events, including logon successes and failures, folder access tracking, and privilege assignments. Managed by security audit policies.
- **System Log**: Records events logged by Windows system components, including driver errors, service startup failures, and hardware alerts.
- **Applications and Services Logs**: Dedicated logs for specific Microsoft components (e.g., PowerShell, Group Policy) and third-party software.

### 2. Event Severity Levels
- **Information**: Indicates successful operation of a service or task (e.g., a service started successfully).
- **Warning**: Indicates a potential problem or non-critical event that may cause issues in the future (e.g., low disk space).
- **Error**: Indicates a significant problem that may cause data loss or service degradation (e.g., service failed to start).
- **Critical**: Indicates a severe failure that caused a system crash or sudden shutdown (e.g., power loss).

### 3. Key Event IDs Reference Table
A support engineer must memorize these key Event IDs for quick diagnostics:

| Event ID | Log Channel | Severity | Meaning | Diagnostic Focus |
|---|---|---|---|---|
| **`41`** | System | Critical | Kernel-Power (Shutdown without clean power-off) | PSU failure, overheating, user held power button |
| **`1014`** | System | Warning | DNS Name Resolution Timeout | Local network drops, DNS server offline |
| **`1074`** | System | Information | System Shutdown/Restart Initiated | Find which user or process triggered a restart |
| **`4624`** | Security | Success Audit | Successful account logon | Verify user authentication and logon type (RDP/Console) |
| **`4625`** | Security | Failure Audit | Failed account logon | Identify password hacking attempts or expired credentials |
| **`4740`** | Security | Success Audit | User account locked out | Find the source machine locking the AD account |
| **`6008`** | System | Error | Unexpected shutdown | Crash, BSOD, or power failure |
| **`7045`** | System | Information | A new service was installed in the system | Auditing unauthorized software installations |

## Commands & Syntax

### PowerShell
```powershell
# Query the local Security log for the 5 most recent failed login attempts (Event 4625)
Get-WinEvent -FilterHashtable @{LogName='Security'; Id=4625} -MaxEvents 5 |
    Select-Object TimeCreated, Message |
    Format-List

# Retrieve unexpected shutdown events (Event 6008) from the System log
Get-WinEvent -FilterHashtable @{LogName='System'; Id=6008} -MaxEvents 10 |
    Select-Object TimeCreated, Message
```

### CMD / Run Box
```cmd
REM Launch the Event Viewer GUI console directly
eventvwr.msc
REM Clear all System event logs from the command prompt
wevtutil cl System
```

### GUI Path
> Start -> search `Event Viewer` -> Expand **Windows Logs** -> Select target channel (Application, Security, System).

### Important Registry Paths
```
HKLM\SYSTEM\CurrentControlSet\Services\EventLog\Application\
HKLM\SYSTEM\CurrentControlSet\Services\EventLog\System\
```

## Real-World Scenarios

### Scenario 1: Identify Which User or Process Restarted a Production Server
**User Complaint:** "The application server rebooted unexpectedly in the middle of the day, disconnecting all active database sessions."
**Your First 3 Checks:**
1. Check the System log for Event ID 1074 (User initiated reboot).
2. Check the System log for Event ID 41 or 6008 (Unexpected shutdown/power loss).
3. Check the Application log for application crash events.
**Diagnosis Steps:**
1. Open Event Viewer -> Navigate to `System` log.
2. Click **Filter Current Log...** -> In the Event ID field, type `1074`.
3. Select the most recent log matching the shutdown timestamp:
   - Message: `The process C:\Windows\system32\shutdown.exe (SERVER01) has initiated the restart of computer SERVER01 on behalf of user Domain\Administrator for the following reason: Operating System: Upgrade (Planned).`
**Root Cause:** A domain administrator manually initiated a restart via `shutdown.exe` to apply updates, without coordinating with the database team.
**Fix:** Update the IT change management process to require scheduled maintenance windows for server restarts.
**Prevention:** Restrict local shutdown privileges on production servers using Group Policy.
**Ticket Close Note:** "Identified shutdown cause: Domain\Administrator initiated planned restart via shutdown.exe (Event ID 1074) to apply updates. Re-aligned admin team with change management policies. Closing ticket."

### Scenario 2: Active Directory Account Kept Locking Out (Event ID 4740)
**User Complaint:** "My Active Directory account keeps locking out every 10 minutes. I unlock it in AD, but it locks out again almost immediately."
**Your First 3 Checks:**
1. Check which Domain Controller authenticated the user.
2. Query the Security log on the DC for Event ID 4740.
3. Check the user's mobile devices and laptops for cached expired credentials.
**Diagnosis Steps:**
1. Log into the Domain Controller. Open Event Viewer -> Security log.
2. Filter the log for Event ID `4740` (Account Lockout).
3. Inspect the event description:
   - Message: `A member account was locked out. Subject: User: jdoe. Caller Computer Name: WORKSTATION-05.`
   - This tells us the bad password requests are originating from `WORKSTATION-05`.
**Root Cause:** The user changed their AD password recently, but their secondary workstation `WORKSTATION-05` was left running with an active script using cached expired credentials, triggering lockout thresholds.
**Fix:** Locate `WORKSTATION-05`, log off the user session or update the cached credentials, and unlock the AD account.
**Prevention:** Configure group policies to terminate stale disconnected sessions after 2 hours.
**Ticket Close Note:** "Tracked lockout source to WORKSTATION-05 using Event ID 4740 on the DC. Logged off the disconnected session. Unlocked user account. Verified no further lockouts. Closing ticket."

## Critical Points

> [!danger] Never Do This
> - Clear system or security event logs using command line utilities (`wevtutil cl`) during an active security incident or audit investigation.
> - Clearing logs destroys forensic evidence, making it impossible to trace the attack vector or identify compromised accounts.
> - Export and archive logs to secure locations before performing maintenance.

> [!warning] Common Trap
> - Filtering by Event ID 41 (Kernel-Power) and immediately assuming the power supply unit (SMPS) is faulty.
> - Event ID 41 simply means the system rebooted without a clean shutdown sequence. This can be caused by a user holding the physical power button, a BSOD crash, or overheating, not just power loss.
> - Verify corresponding Event ID 1001 (Bugcheck) or Event ID 6008 logs to find the cause.

> [!tip] Senior Engineer Tip
> - Create a **Custom View** in Event Viewer that filters for critical Event IDs across System and Application channels (e.g., `41, 1001, 6008, 7045`). Save this view as an XML file, allowing you to import it to any client system for quick diagnostics.

> [!success] Verification Steps
> - Run the command: `Get-WinEvent -FilterHashtable @{LogName='System'; Id=6008} -ErrorAction SilentlyContinue` to verify unexpected shutdowns.
> - Check if recent timestamps match client reports.

> [!question] Interview Alert
> - "How do you identify the source computer causing a user's Active Directory account to lock out?"
> - Answer: "I log into a Domain Controller, open the Event Viewer Security log, and filter for Event ID 4740. I select the lockout event for the user, and inspect the description to find the 'Caller Computer Name', which reveals the hostname of the workstation sending the bad credentials."

## Common Mistakes & Fixes
| Mistake | Why It Happens | Correct Approach |
|---------|----------------|------------------|
| Assuming Event 41 is a bad PSU | Ignoring corresponding crash dumps | Check for Event 1001 (Bugcheck) to verify if a BSOD occurred before the power drop. |
| Forgetting to enable logon auditing | Security log is empty of logon events | Enable **Audit Logon Events** in the Default Domain Controllers GPO. |
| Storing massive log file sizes | Log buffer allocation exhaustion | Configure logs to **Overwrite events as needed** and set maximum size limits. |

## Lab Exercise

**Objective:** Create a Custom XML View in Event Viewer filtering for critical system restart and crash events, export the view configuration, and query logs using PowerShell.
**Time Required:** 20 minutes
**Environment Needed:** A Windows 11 VM or physical machine with local administrator rights.
**Pre-requisites:** Administrative access.

**Steps:**
1. Open Event Viewer: Press `Win + R` -> type `eventvwr.msc`.
2. Click **Create Custom View...** in the right-hand Actions pane.
3. Configure Filter:
   - Logged: **Any time**
   - Event level: Check **Critical**, **Warning**, and **Error**
   - By log: Select **Windows Logs** -> Check **Application** and **System**
   - Event IDs: Type `41, 1014, 1074, 6008, 7000`
4. Click **OK** -> Name the custom view: `Critical System Events` -> Click **OK**.
   - Expected output: The new custom view displays in the left pane, showing filtered events.
5. Export Custom View: Right-click `Critical System Events` -> Select **Export Custom View...** -> Save as `SystemFilter.xml`.
6. Open PowerShell as Administrator. Run a query using the same Event IDs:
   ```powershell
   Get-WinEvent -FilterHashtable @{LogName='System'; Id=1074,6008} -MaxEvents 5
   ```
7. Verification: Confirm the PowerShell output displays the correct event details.

**Success Criteria:** The custom view filters the correct events, the XML file is successfully exported, and the PowerShell query returns the logs.
**Common Failures:** "No events found" error if the VM has a clean install with no historical events.

## Interview Questions & Answers

### Basic (L1 Level)
**Q: What is the difference between the Application, System, and Security logs in Event Viewer?**
A: The Application log records events generated by local programs (e.g., application crashes). The System log records events generated by Windows operating system components (e.g., driver errors or service failures). The Security log records security audit events (e.g., successful and failed logins).

**Q: What does Event ID 41 mean, and what are its common causes?**
A: Event ID 41 (Kernel-Power) indicates the system rebooted without performing a clean shutdown sequence. Common causes include a user holding down the physical power button, a power supply failure, or a system thermal shutdown caused by overheating.

### Intermediate (L2 Level)
**Q: What is Event Forwarding, and how do you configure it in a Windows environment?**
A: Event Forwarding allows administrators to collect event logs from multiple workstations and consolidate them onto a centralized Collector server. To configure it, I enable WS-Management (`WinRM`) on the client systems, start the Windows Event Collector service on the collector, and configure a **Subscription** defining the log channels and target computers.

### Advanced (L3/Senior Level)
**Q: A domain-joined computer is suspected of being compromised. The security team requests a list of all successful interactive logins and service installations over the last 24 hours. Explain your extraction process.**
A:
- **Situation**: A workstation was suspected of compromise, requiring a rapid audit of logon events and service changes.
- **Task**: I needed to query the logs for Event 4624 (Logon) and Event 7045 (Service Install).
- **Action**: I wrote a PowerShell script utilizing `Get-WinEvent` with a filter hash table targeting the Security and System logs over the last 24 hours:
   `$Date = (Get-Date).AddDays(-1)`
   `Get-WinEvent -FilterHashtable @{LogName='Security'; Id=4624; StartTime=$Date}`
   `Get-WinEvent -FilterHashtable @{LogName='System'; Id=7045; StartTime=$Date}`
- **Result**: The script generated a CSV report detailing the logon accounts (filtered by Logon Type 2 for console and Type 10 for RDP) and newly installed services, which I provided to the security team.

### HR / Behavioral
**Q: Tell me about a time you used event logs to identify a difficult application issue that developer teams could not locate.**
A: Users reported an application crash error on launch. The developers insisted the application code was correct. I checked the Application log, found Event ID 1000 pointing to a missing `.NET Framework` runtime DLL, installed the runtime, and resolved the issue.
